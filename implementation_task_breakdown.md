# 総合商社シミュレーション 実装タスク分解

## 1. このドキュメントの役割
既存の仕様書を、実装着手できるレベルのバックログに分解したもの。
対象は「最初に面白さを検証する縦切りプロトタイプ」と「初期MVP」。

前提として、初期プロトタイプは以下で進める。
- クライアント完結型のシングルプレイ
- データはローカル JSON
- 実装言語は TypeScript 想定
- セーブはローカル保存
- AI対戦、ネットワーク、演出強化は後回し

## 2. 開発の基本方針
- 最初に作るのは「全部入り」ではなく「断片をつないで案件化する快感」の縦切り
- UI より先に、ターン進行・案件組成・交渉判定の中核ロジックを固定する
- コンテンツ量で面白さをごまかさず、3案件テンプレートで成立するかを先に検証する
- 交渉、稟議、リスクはそれぞれ独立実装ではなく、同じ案件オブジェクトを更新する形で統一する

## 3. マイルストーン

| Milestone | 目的 | 完了条件 |
| --- | --- | --- |
| M0 | 実装基盤を作る | ターン進行とゲーム状態保存ができる |
| M1 | 商社らしい縦切りを成立させる | 断片情報から1件の案件を組成し、交渉して承認まで進められる |
| M2 | 運営と危機対応を入れる | 実行中案件が収益化し、イベント対応で結果が変わる |
| M3 | ポートフォリオとして評価できる | 2件以上の案件でシナジーと集中リスクが見える |
| M4 | 初期MVPとして通しプレイ可能にする | 6-8ターンを最後まで遊べて結果評価が出る |

## 4. 推奨実装順

1. ゲーム状態、ターン進行、決算
2. 断片情報生成、案件リード生成
3. 案件組成キャンバス、成立スコア
4. 交渉
5. 社内稟議
6. 実行中案件、収益化
7. リスクイベント
8. シナジー、ポートフォリオ評価
9. チュートリアル、難易度、コンテンツ拡張

## 5. 依存関係の考え方

### 5-2. UI より先に必要なロジック

- 稟議判定
- リスク適用順

## 6. 実装バックログ

### T01. コア定数と型の土台を作る
- 目的: 全システムで共通利用する enum、タグ、ID ルールを固定する
- 主な内容:
  - 地域、産業、関係者種別、案件状態、イベント種別の列挙
  - 案件スロット種別の列挙
  - タグ命名規約
  - ID 命名規約
- 成果物:
  - `types`
  - `constants`
  - `content schema`
- 受け入れ条件:
  - 主要データが型エラーなく参照できる
  - 断片情報、案件、パートナーが同じタグ体系で接続できる

### T02. ゲーム状態ストアを作る
- 目的: 1つの状態ツリーでゲーム全体を管理できるようにする
- 主な内容:
  - `GameState` の初期化
  - ターン、フェーズ、プレイヤー、保有案件、イベント履歴の保存
  - ローカルセーブ/ロード
- 依存:
  - T01
- 受け入れ条件:
  - 新規ゲーム開始、途中保存、再開ができる
  - 状態破損なくターンをまたげる

### T03. ターン進行エンジンを実装する
- 目的: フェーズ順に処理が進む骨格を作る
- 主な内容:
  - 情勢変化
  - 情報収集
  - 案件発見
  - 案件組成
  - 交渉
  - 稟議
  - 実行/運営
  - リスク対応
  - 決算
- 依存:
  - T02
- 受け入れ条件:
  - 任意のターンでフェーズが順番に遷移する
  - フェーズスキップ、戻る、確定のルールが定義される

### T04. プレイヤーリソース計算を実装する
- 目的: キャッシュ、信用、社内評価、人材などの基本リソースを処理する
- 主な内容:
  - 年初リソース付与
  - アクション消費
  - 年末のリソース更新
  - 破綻条件チェック
- 依存:
  - T02
- 受け入れ条件:
  - 行動によって資源が増減する
  - 資本不足や信用悪化が翌年に効く

### T05. マクロ市場更新ロジックを実装する
- 目的: 市況に応じて案件価値が毎ターン変わる状態を作る
- 主な内容:
  - 地域別需要成長更新
  - 為替更新
  - 資源価格更新
  - 規制ムード更新
- 依存:
  - T03
- 受け入れ条件:
  - 毎ターンの市場条件が変動する
  - 変動結果が断片情報生成や案件採算に反映される

### T06. 断片情報生成システムを実装する
- 目的: 世界の断片が毎ターン出現する状態を作る
- 主な内容:
  - 地域 x 産業 x 情報源ごとの抽選
  - 情報の確度、期限、タグ付与
  - 情報収集人材による補正
- 依存:
  - T04
  - T05
- 受け入れ条件:
  - 1ターンで複数の断片情報が生成される
  - 情報源や配置人材によって質が変わる

### T07. 案件リード生成ルールを実装する
- 目的: 断片情報の組み合わせから仮説案件が開く状態を作る
- 主な内容:
  - Need/Supply/Finance/Access の組み合わせ検知
  - リードスコア算出
  - リード候補提示
- 依存:
  - T06
- 受け入れ条件:
  - プレイヤーが複数の断片を組み合わせるとリードが生成される
  - 同じ断片群から複数テンプレート候補が出る場合がある

### T08. 案件テンプレートとスロット検証を実装する
- 目的: リードを「案件キャンバス」に載せられるようにする
- 主な内容:
  - 案件テンプレート定義
  - 必須スロットの表示
  - 適合断片/パートナーのバリデーション
- 依存:
  - T01
  - T07
- 受け入れ条件:
  - 案件ごとに不足条件が可視化される
  - 不適合な要素はスロットに置けない

### T09. 案件成立スコアリングを実装する
- 目的: 案件の完成度が数字で見えるようにする
- 主な内容:
  - スロット充足度
  - 条件適合
  - リスク露出
  - 資金成立性
  - シナジー補正
- 依存:
  - T08
- 受け入れ条件:
  - スロット変更で成立スコアと IRR が変わる
  - 不成立状態では稟議に進めない

### T10. パートナー名簿と候補選定を実装する
- 目的: スロットを埋める相手を選ぶ楽しさを作る
- 主な内容:
  - パートナーマスタ
  - 地域/産業適性
  - 関係値
  - red line
  - offerable slot
- 依存:
  - T01
- 受け入れ条件:
  - 同じスロットでも相手によって成立率や収益率が変わる
  - 関係値が不足すると候補が出ない場合がある

### T11. 交渉システムを実装する
- 目的: 相手の目的関数を読むゲーム性を作る
- 主な内容:
  - 議題選択
  - 提案カード
  - 譲歩
  - hidden weight
  - 妥結/保留/決裂判定
- 依存:
  - T09
  - T10
- 受け入れ条件:
  - 1回の交渉結果で契約条件と関係値が更新される
  - 相手ごとの重視項目が体感できる

### T12. 社内稟議システムを実装する
- 目的: 商社らしい社内調整の壁を作る
- 主な内容:
  - 財務、法務、リスク、ESG、経営陣の評価計算
  - veto 条件
  - 根回しアクション
  - 条件付き承認
- 依存:
  - T09
  - T11
- 受け入れ条件:
  - 外部条件が良くても、社内理由で止まる案件がある
  - 根回しや条件修正で通るようになる

### T13. 建設・立ち上げ・稼働ロジックを実装する
- 目的: 承認後の案件が事業として育つ状態を作る
- 主な内容:
  - 実行中案件の進捗管理
  - 収益化タイミング
  - 稼働率
  - コスト超過
  - オペレーション改善
- 依存:
  - T12
- 受け入れ条件:
  - 案件が年をまたいで状態遷移する
  - 人材再配置や追加投資で結果が変わる

### T14. リスクイベントエンジンを実装する
- 目的: 危機対応の選択が価値を持つようにする
- 主な内容:
  - 発生条件判定
  - 露出タグとの照合
  - 対応選択肢表示
  - 影響反映
- 依存:
  - T05
  - T13
- 受け入れ条件:
  - 市況、地政学、物流、ESG の最低4カテゴリが動く
  - 対応によって短期損益と長期構造が変わる

### T15. シナジー判定とポートフォリオ評価を実装する
- 目的: 単独案件でなく全社最適を評価できるようにする
- 主な内容:
  - バリューチェーン接続判定
  - 地域プラットフォーム判定
  - 集中リスク罰則
  - 企業価値スコア算出
- 依存:
  - T13
  - T14
- 受け入れ条件:
  - 2件以上の保有でシナジーが発動する
  - 高収益でも集中しすぎると最終スコアが落ちる

### T16. ワールドマップと共通 HUD を実装する
- 目的: すべての判断を支える共通の状況把握 UI を作る
- 主な内容:
  - 年度表示
  - フェーズ進行
  - キャッシュ
  - 投資余力
  - 社内評価
  - 現地信頼
  - 通知欄
- 依存:
  - T03
  - T04
- 受け入れ条件:
  - どの画面からでも現在地が分かる
  - 主要リソースが常時参照できる

### T17. 情報収集ボード UI を実装する
- 目的: 人材配分と断片情報獲得を見た目で理解できるようにする
- 主な内容:
  - 地域 x 産業マトリクス
  - 人材ドラッグ配置
  - 見込み獲得情報のプレビュー
- 依存:
  - T06
  - T16
- 受け入れ条件:
  - 配置変更で期待結果が変わることが分かる

### T18. 案件インボックスと案件組成キャンバス UI を実装する
- 目的: このゲームの一番おいしい部分を playable にする
- 主な内容:
  - 断片情報一覧
  - リード候補
  - 案件テンプレート選択
  - スロットへの drag and drop
  - 収益/リスク/承認難易度の変化表示
- 依存:
  - T07
  - T08
  - T09
- 受け入れ条件:
  - プレイヤーが UI 上で案件を1件組成できる
  - 何が足りないのか一目で分かる

### T19. 交渉画面 UI を実装する
- 目的: 提案と譲歩の結果が分かる会話型画面を作る
- 主な内容:
  - 相手情報
  - 現在提案
  - 推定ニーズ
  - 受諾度表示
  - 交渉ログ
- 依存:
  - T11
- 受け入れ条件:
  - 1相手に対して最大3ラウンド交渉できる
  - 結果が案件条件に反映される

### T20. 稟議画面 UI を実装する
- 目的: 社内政治を明確に見せる
- 主な内容:
  - 部門別賛否メーター

- 依存:
  - T12

  - なぜ通らないかが分かる
  - 再提出の余地が可視化される

### T21. ポートフォリオ/決算 UI を実装する

## 9. プレイテスト時の確認項目
- 断片情報を集める意味が分かるか
- 案件組成の途中で「何を足せば成立するか」が理解できるか
- 交渉相手ごとの違いを感じるか
- 社内稟議が理不尽でなく、悩みとして成立しているか
- 高利益案件より、次につながる案件を選ぶ動機が出るか
- 危機対応が罰ゲームではなく「再設計の見せ場」になっているか

## 10. この順番で作る理由
- 交渉や稟議を先に UI だけで作ると、意味の薄いミニゲームになりやすい
- 先に案件オブジェクトとスコアリングを固定すると、後続の全画面がつながる
- シナジーやポートフォリオは魅力的だが、最初に入れると複雑さが勝つ
- まずは「断片をつなぐ」「相手を動かす」「社内を通す」の3点を先に成立させる
